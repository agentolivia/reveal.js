CEM Class Syntax

aka (BEM)

drupal.org/coding-standards/css/architecture

The Rules

  • We use classes; ID's are the devil
  • Components (The C in CEM) are a body of markup that works together;
    e.g. a menu, a teaser, a who's online widget, a social share widget, an article.
  • An element (the E in CEM) is a piece of the whole.
    e.g. a title, chunk of text, an image, call to action link.
  • A modifier is a suffix that can denote a slightly different version of a Component or Element
  • Names are '-' delimited (this goes for a C, E or M) 
    .component-name {}
  • Element classes have the component's name, then their name with a double underscore separator 
    .component-name__element-name {}
  • Modifiers are added to the end of a component or an element and are separated by a dash. 
    
.component-name--modifier-name {}
.component-name__element-name--modifier-name {}

Menu Example


.menu__wrapper {}
.menu {}
  .menu__list {}
    .menu__list-item {}
    .menu__link {}
    .menu__link.menu__link--active {}
    .menu__link {}
      .menu.menu--sub-menu {}
        .menu__list {}

JS Prefix

If a class is coming from JS, it gets a "js-" prefix.


.js-open {}
.js-accordion {}
.js-best-thing-you-ever-saw {}

Pro-tip*: Utility classes


.u-button {}
.u-clearfix {}
.u-text-replace {}
.u-element-invisible {}

* This is something we do at Lullabot on our BEM projects,
and not something core is doing.

When you're using BEM, sometimes you end up having great use cases for a few utility classes, instead of repeating styles. Prefix them with a "u-" so they're easy to spot.

The blessing and curse of Twig's dot notation

Dot notation on twig vars will check the following things, in this order, and give you back the result of the first on it hits:


  /**
   * If you have {{ sandwhich.cheese }} in your twig, it checks:
   */
  // Array key.
  $sandwich['cheese'];
  // Object property.
  $sandwich->cheese;
  // Also works for magic get (provided you implement magic isset).
  $sandwich->__isset('cheese'); && $sandwich->__get('cheese');
  // Object method.
  $sandwich->cheese();
  // Object get method convention.
  $sandwich->getCheese();
  // Object is method convention.
  $sandwich->isCheese();
  // Method doesn't exist/dynamic method.
  $sandwich->__call('cheese');

In other words, it could be getting data, or performing functionality. I find this to be helpful AND confusing at the same time. Helpful if you know what it’s doing, confusing if you’re not sure what’s going on.

Array Filters


  {# Without will omit part of the array, this is being used instead of hide()/show() #}
  {{ node|without('comments') }}

  {# Get the keys of an array (as an array) #}
  {% set my_array_keys = my_array|keys %}

  {# Append an array's values to another #}
  {% set fruits_and_veggies_array = fruits_array|merge(veggies_array) %}

  {# Reverse the values %}
  {% set countdown = [1, 2, 3, 4, 5]|reverse %}

Devel & Kint

Last I've checked dpm, dsm, kpr aren't the best go to when dumping vars. They can hit some recursion and peak out your memory.

8.x.x Devel has a new submodule called Devel Kint, enable Kint and use kint() instead.

Getting into the weeds

No Theme Functions*

(*)There are a few. Link is one of them, I'm having a hard time tracking down the others.

This is for performance reasons, the few are very tiny and commonly used elements with a lot of logic. The Drupal Twig peoples don't think we'll be able to get rid of them in Core.

BUT you can get around using them in your theme templates. (example?)

But still Render Arrays

Because the render array's theme hook (aka #theme) now render with twig templates, instead of theme functions.


  // In Preprocess
  $variables['table'] = array(
    '#theme' => 'table',
    '#header' => $header,
    '#rows' => $rows,
  );

By default will render using the table.html.twig template file (theme_suggestions can override this).

Speaking of theme_suggestions...

They're all done in one place, instead of in whatever preprocessor happens to preprocess them!

See hook_theme_suggestions_alter(array &$suggestions, array $variables, $hook)

https://api.drupal.org/api/drupal/core%21lib%21Drupal%21Core%21Render%21theme.api.php/function/hook_theme_suggestions_alter/8

Attribues Object, your new friend!


  <div{{ attributes }}>

No extra space needed!


  {#  ▼ Don't do this #}
  <div {{ attributes }}>

https://www.drupal.org/node/2513632

Has all these cool methods!


  {# Add or Remove Classes #}
  <div{{attributes.addClass('my-class')}}>
  <div{{attributes.removeClass('their-class')}}>

  {# Set any attribute #}
  <div{{ attributes.setAttribute('id', 'myID') }}>
  <div{{ attributes.setAttribute('data-bundle', node.bundle) }}>

  {# Remove any attribute #}
  <div{{ attributes.removeAttribute('id') }}>

  {# hasClass boolean! #}
  {% if attribute.hasClass(‘myClass') %}
   {# do stuff #}
  {% endif %}

https://www.drupal.org/node/2513632

Attributes methods can be chained


  <div{{ attributes.addClass('hello').removeClass('goodbye')}}>

Best practices encourage a bit more logic in the template files

My favorite update is setting more classes in the template


  {%
    set classes = [
      'node--'  ~ node.bundle|clean_class,
      'node--my-custom-modifier',
    ]
  %}

  <article{{ attributes.addClass(classes) }}>
